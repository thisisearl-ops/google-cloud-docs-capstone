Overview

Agent Platform uses IAM to manage access to resources. When you plan access control for your resources, you can manage access at the project level or at the resource level.

Project-level access applies to all resources in that project. Resource-level access applies only to a specific resource.

You grant access by assigning IAM roles to principals. Predefined roles make setup easier, but custom roles are recommended when you want to limit access to only the permissions that are required.

IAM roles

Agent Platform supports three common IAM role approaches:

  • Custom roles let you choose a specific set of permissions, create your own role with those permissions, and grant the role to users in your organization.
  • Predefined roles grant a set of related permissions to Agent Platform resources at the project level.
  • Basic roles, such as Owner, Editor, and Viewer, provide broad project-level access and are common to all Google Cloud services.

To add, update, or remove these roles in your Agent Platform project, use the IAM workflow for granting, changing, and revoking access.

Use custom roles for least privilege

Predefined roles often contain more permissions than you need. You can create custom roles to grant principals only the specific permissions required for a task.

For example, you can create a custom role with the aiplatform.endpoints.predict permission, and then assign that role to a service account on an endpoint. This lets the service account call the endpoint for predictions without granting broader endpoint control.

Basic roles

The older Google Cloud basic roles are common to all Google Cloud services. These roles are Owner, Editor, and Viewer.

Basic roles provide broad access at the project level. For Agent Platform, use predefined or custom roles when you need more focused permissions.

Project-level versus resource-level access

You can manage access at the project level or resource level. You might also have the ability to manage access at a folder or organization level.

For most Agent Platform resources, access can only be controlled by the project, folder, and organization. Access to individual resources can be granted only for specific resource types, such as an endpoint or a featurestore.

Setting a policy at the resource level does not affect project-level policies. A resource inherits policies from its ancestry, so you can combine broad read permissions at the project level with more specific write permissions on individual resources.

Supported resources

Agent Platform supports resource-level access for Vertex AI Feature Store featurestore resources, entity type resources, and Model Registry resources.

Not all Agent Platform predefined roles and resources support resource-level policies. After granting or revoking access to a resource, changes can take time to propagate.

Resources, service accounts, and service agents

Agent Platform services often manage long-running resources that perform actions, such as running training jobs that read data or serving ML models that read model weights. These standalone resources have their own resource identity when performing actions.

This identity is distinct from the principal that created the resource. Permissions granted to the resource identity define which data and other resources that identity can access.

By default, Agent Platform resources use service accounts managed by Agent Platform as a resource identity. For some services, you can specify a service account to attach to the resource.

Service accounts

A service account is a special account used by an application or a virtual machine instance, not a person. You can create and assign permissions to service accounts to provide specific permissions to a resource or application.

For custom training containers or containers that serve online predictions for custom-trained models, a custom service account can be used to control which permissions are available to the workload.

Service accounts are identified by an email address.

Service agents

Service agents are automatically provided accounts that let Google Cloud services access resources on your behalf.

When a service agent is created, it is granted a predefined role for your project. The complete table is long, so this prototype shows the most relevant Agent Platform service agents.

The Gemini Enterprise Agent Platform Custom Code Service Agent is created only if you run custom training code to train a custom-trained model. In some cases, you must grant additional roles to a service agent, such as when Agent Platform needs to access a Cloud Storage bucket or BigQuery table in a different project.

Grant Agent Platform service agents access to other resources

Sometimes you need to grant additional roles to an Agent Platform service agent. For example, if Agent Platform needs to access a Cloud Storage bucket in a different project, you must grant one or more additional roles to the service agent.

Role addition requirements for Cloud Storage

If you access data in a Cloud Storage bucket in a different project, grant Agent Platform the Storage Object Viewer role in that project.

If you use a Cloud Storage bucket to receive data from your local computer for an import operation, and the bucket is in a different project than your Google Cloud project, grant Agent Platform the Storage Object Creator role in that project.

Grant access in your home project

To grant additional roles to a service agent for Agent Platform in your home project:

  • Go to the IAM page of the Google Cloud console for your home project.
  • Select the Include Google-provided role grants checkbox.
  • Determine the service agent you want to grant permissions to and click the pencil icon.
  • Filter for Principal:@gcp-sa-aiplatform-cc.iam.gserviceaccount.com to find Agent Platform service agents.
  • Grant the required roles to the service agent and save your changes.
Grant access in a different project

When you use data sources or destinations in a different project, you must give the Agent Platform service agent permissions in that project. The service agent is created after you start the first asynchronous job, such as creating an endpoint. You can also explicitly create the Agent Platform service agent with gcloud beta services identity create.

  • Go to the IAM page for your home project, where you are using Agent Platform.
  • Select the Include Google-provided role grants checkbox.
  • Find the service agent and copy its email address from the Principal column.
  • Change projects to the project where permissions are needed.
  • Click Add, enter the service agent email address in New principals, add all required roles, and save.
Provide access to Google Sheets

If you use an external BigQuery data source backed by Google Sheets, share your sheet with the Agent Platform service account. The service account is created after the first asynchronous job, or you can explicitly create it with the Google Cloud CLI.

  • Go to the IAM page of the Google Cloud console.
  • Look for the service account named Agent Platform Service Agent and copy its email address from the Principal column.
  • Open your Sheets file and share it with that address.
